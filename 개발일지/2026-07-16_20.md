## 신규 기능 — 도메인별 검색

### 배경

기존엔 질문을 보내면 전체 문서에서 근거를 찾았다. 인사·복지, 정보화·보안 등 **분야를 좁히면 정확도가 올라가므로**, 사용자가 검색 범위를 도메인으로 한정할 수 있게 했다.

### 구현

- `GET /api/v1/capabilities` 신규 연동 — 서버가 주는 도메인 목록(code, label)을 진짜 소스로 사용
  - `types/capability.ts`, `api/services/capability.ts`, `hooks/useCapabilities.ts` 추가
- 채팅 입력창에 `DomainPicker` 드롭다운 추가 (`ui/components/chat/DomainPicker.tsx`, `DomainIcon.tsx`)
- 선택한 도메인을 스트리밍 요청의 `domain` 필드로 전송 (`StreamMessageRequest`에 `domain` 추가)
- 보낸 질문 말풍선 위에 도메인 태그 표시 (`MessageRow.tsx`)

### 핵심 설계 — "없는 도메인은 안 뜨게"

capabilities는 정의된 도메인 **전체**를 주지만, 실제로 그 도메인의 문서가 있는지는 별개였다. 그래서 capabilities 목록과 실제 문서 목록의 `domain` 값을 **교집합**해서, 문서가 존재하는 도메인만 드롭다운에 노출하도록 했다.

> **핵심 교훈:** API가 "가능한 값 전체"를 준다고 그대로 노출하면 안 된다.
> "실제 데이터에 존재하는 것"과 교집합해야 사용자에게 유효한 선택지만 보여줄 수 있다.

## UX — 로딩 깜빡임 방지

교집합에는 capabilities와 문서 목록 두 API가 모두 필요한데, 각자 따로 로딩돼서 한쪽만 먼저 도착하면 도메인 버튼이 **잠깐 사라졌다 다시 뜨는** 깜빡임이 있었다.
→ 두 로딩이 모두 끝나기 전까지는 스켈레톤(회색 자리표시)으로 자리를 잡아, 목록이 나중에 채워질 때 레이아웃이 흔들리지 않게 했다.

## 디자인 — 입력창 레이아웃 정리

- 첨부·전송·도메인 선택 버튼의 높이(40px)·아이콘 크기(20px)를 통일해 들쑥날쑥하던 하단 바를 정렬
- 드롭다운이 입력창의 `overflow-hidden`에 잘리던 문제 → `position: fixed`로 뷰포트에 띄워 해결
- 입력창 하단 빈 여백을 클릭해도 텍스트에 포커스가 가도록 처리

## 정확성 버그 — 도메인 정보의 상태 동기화

이번 작업에서 가장 많은 버그가 나온 지점. 도메인 정보가 **저장·전송·복원되는 경로가 여러 갈래**여서, 한 곳만 빠뜨려도 유실됐다.

### 버그 1 — 새 채팅 첫 메시지가 '전체'로 전송됨

**문제:** `ChatInput`이 도메인을 포함해 저장한 inflight를, `ChatPage`가 도착 직후 `saveInflight(sessionId, initialMessage)`로 **도메인 없이 덮어썼다.** `ChatPage`는 질문 텍스트만 넘겨받고 도메인은 몰랐기 때문.

```ts
// ChatPage.tsx (버그) — 도메인 없이 재저장 → 방금 저장한 도메인을 날림
if (initialMessage) {
  saveInflight(sessionId, initialMessage);
}
```

**해결:** `ChatPage`의 중복 `saveInflight` 호출을 제거. `ChatInput`이 navigate 직전에 이미 도메인까지 저장하므로 재저장 자체가 불필요했다.

> **핵심 교훈:** 같은 값을 두 지점에서 저장하면, 정보가 적은 쪽이 많은 쪽을 덮어쓰는 사고가 난다. 저장 책임은 한 곳으로 모은다.

### 버그 2 — 같은 질문에 엉뚱한 도메인 태그가 붙음

**문제:** 세션 재진입 시 서버 메시지엔 도메인이 없어(백엔드 미저장), 로컬 캐시에서 도메인을 복원했다. 그런데 **질문 내용(content)으로 매칭**해서, 같은 "안녕"이 세션 A(인사·복지)·세션 B(전체)에 있으면 B의 "안녕"에도 인사·복지가 잘못 붙었다.

```ts
// (버그) 내용이 같으면 무조건 같은 도메인이라고 가정
domainByContent.set(m.content, { code: m.domainCode, label: m.domainLabel });
```

**해결:** content 매칭을 버리고 **순서(index) 기반**으로 변경. 한 세션 내 user 메시지의 순서는 캐시와 서버가 동일하므로, 순서대로 짝지어 도메인만 이식한다. 캐시·서버의 user 메시지 개수가 다르면(어긋날 위험) 이식을 아예 건너뛴다.

> **핵심 교훈:** 매칭 키로 "내용"을 쓰면 중복 데이터에서 오염된다. 위치·순서·ID 같은 **식별 가능한 키**를 써야 한다.

### 버그 3 — 스트리밍 중 새로고침 시 도메인 유실

**문제:** `sendMessage`가 `saveInflight`에 도메인을 안 넘겼다(`retryLastMessage`는 넘기는데 비대칭). 답변 생성 도중 새로고침하면 inflight로 복구될 때 '전체'로 떨어졌다.

**해결:** `sendMessage`도 `saveInflight(sessionId, content, domain)`으로 도메인을 함께 저장.

> **핵심 교훈:** 같은 정보를 다루는 함수가 둘 이상이면, 한쪽만 고치고 다른 쪽을 빠뜨리기 쉽다. 비대칭이 곧 버그다.

### 저장소 선택 — sessionStorage vs localStorage

- **대화 태그(과거 기록):** `localStorage` — 탭을 닫았다 열어도 유지 (같은 브라우저 한정)
- **입력창의 도메인 선택(임시 상태):** `sessionStorage` — 탭 닫으면 리셋. 오래된 선택이 새 세션에 몰래 걸려있는 함정을 방지

> **핵심 교훈:** "오래 유지되니 localStorage가 무조건 낫다"는 틀렸다. "유지되면 사용자가 반길 값인가, 당황할 값인가"로 저장소를 고른다.

> **남은 한계:** localStorage도 이 브라우저 한정이라, 다른 기기·시크릿 모드에선 태그가 안 뜬다. 완전한 해결은 백엔드가 메시지에 도메인을 저장·반환해야 한다.

## UX 버그 — 진입 자동 스크롤이 계속 내려감

**문제:** 세션 진입 시 최하단으로 내리는 자동 스크롤이 "한 번만" 하려 했으나, 첫 스크롤 발동 전에 `messages`가 여러 번 갱신(캐시 복원·도메인 이식 등)되면서, 사용자가 위로 올려도 뒤늦게 도로 내려갔다.

**해결:** "사용자가 먼저 스크롤하면 자동 하단 이동을 포기"하는 플래그(`userScrolledBeforeAnchor`)를 추가. 채팅 스크롤의 표준 패턴이다.

> **핵심 교훈:** 자동 스크롤은 "사용자 의도"를 항상 우선해야 한다. 사용자가 스크롤한 순간 자동 동작을 멈추는 게 원칙.

## 이번 기간 회고

- **도메인 하나를 화면 끝까지 전달하는 데 경로가 4~5군데**였고(입력창 → inflight → store → 메시지 → 캐시 → 복원), 그중 한 곳만 빠져도 유실됐다. 상태가 여러 저장소·여러 함수를 거칠 때는 **경로 전체를 그려놓고 각 지점을 점검**해야 한다.
- 근본 한계(다른 기기에서 태그 유실)는 프론트만으로 못 푼다. **백엔드가 도메인을 저장하도록 요청**하는 것이 정석 해결.
