# 개발 일지 — 스크롤 앵커 버그 추적

**날짜:** 2026.7.22
**대상:** llm-frontend (React + TypeScript)
**요약:** "질문 상단 고정 후 위로 스크롤해도 자동으로 다시 끌려가는" 버그를 추적했다. 세 번의 가설이 전부 빗나가면서 반나절 넘게 파고든 끝에 진짜 트리거(스트림 재시작)를 찾아냈다. 코드 수정까지는 다음 세션으로 넘김.

---

## 진행 중 — 스크롤 앵커 버그 (미해결, 다음 세션 계속)

### 증상

질문을 보내면 그 질문이 화면 상단에 부드럽게 고정된다(정상, 아주 잘 동작하는 기능). 그런데 그 상태에서 **사용자가 위로 스크롤해 이전 대화를 보려 하면, 잠시 후 자동으로 다시 아래로 끌려간다.** "다시 끌려가면 안 된다"가 요구사항.

### 가설 1 — 실패: "재실행되는 effect가 scrollTop을 강제 복원한다"

앵커 effect가 스트리밍 토큰마다 재실행되며 spacer(여백)를 재계산하는데, 그 직후 `el.scrollTop = savedTop`으로 스크롤 위치를 복원하는 코드가 있었다. 이게 사용자가 방금 올린 스크롤까지 되돌리는 거라 보고, "앵커된 이후 사용자가 스크롤하면 이 복원을 건너뛴다"는 플래그(`userScrolledAfterAnchor`)를 추가했다.

**결과: 그대로 재현됨.** 디버그 로그로 보니 이 방어 로직 자체는 조건상 맞는데도 여전히 끌려갔다.

### 가설 2 — 실패: "앵커 애니메이션이 스스로를 사용자 스크롤로 오인한다"

`animateAnchor`가 `requestAnimationFrame`으로 매 프레임 `scrollTop`을 코드가 직접 바꾸는데, 브라우저는 코드가 움직인 스크롤과 사용자가 움직인 스크롤을 `scroll` 이벤트로 구분해주지 않는다. 그래서 애니메이션 자체가 "사용자가 방금 스크롤했다"는 플래그를 스스로 세워버리는 게 아닌가 의심하고, `programmaticScrollRef`로 코드발 스크롤을 표시해 걸러내는 로직을 추가했다.

**결과: 여전히 재현됨.** 로그를 찍어보니 `[ANIMATE-ANCHOR-CALLED]`, `[ANCHOR-EFFECT]` 자체가 문제의 그 구간에 **한 번도 안 찍혔다** — 즉 이 effect들은 범인이 아니었다.

> **핵심 교훈:** "이 코드가 원인일 것이다"라는 가설을 세우면, 고치기 전에 **그 코드가 실제로 그 순간 실행되는지부터 로그로 확인**해야 한다. 실행도 안 되는 코드를 아무리 고쳐도 증상은 그대로다.

### 가설 3 — 부분 실패: "CSS Scroll Anchoring(브라우저 내장 기능)"

앵커 관련 effect가 전혀 안 찍히는데도 `scrollTop`이 매끄럽게 계속 증가하는 패턴은, 브라우저가 콘텐츠 높이 변화를 감지해 스스로 스크롤을 보정하는 **네이티브 CSS Scroll Anchoring**과 정확히 일치했다. `overflow-anchor: none`이 Tailwind 임의값 클래스로 이미 걸려 있었지만, 인라인 `style`로 스크롤 컨테이너와 콘텐츠 wrapper 양쪽에 명시적으로 다시 걸었다.

**결과: 여전히 재현됨.**

### 진짜 원인 발견 — 사용자 확인으로 좁혀짐

디버그 로그에 타임스탬프를 붙여 시간순으로 다시 관찰한 결과, 결정적 패턴을 확인했다: **휠을 굴리는 동안엔 안 끌려가다가, 손을 떼고 25초쯤 가만히 있으면 그제서야 끌려간다.** 로그를 보면 그 순간 `[ANCHOR-EFFECT]`에 `isNewQuestion: true`가 찍혔다 — 즉 **스트리밍이 끝난 지 한참 지난 뒤, 어떤 assistant 메시지가 다시 `status: 'streaming'`으로 전환되며 새 스트림처럼 인식**되고 있었다.

사용자에게 재현 조건을 직접 확인한 결과: **질문은 딱 하나만 보냈고, 재생성도 누르지 않았는데 25초 뒤 저절로 새 스트림이 감지됐다.** 즉 버그는 스크롤 앵커 로직이 아니라 **`chatStore`가 스트리밍을 예상치 못하게 재시작시키는 지점**에 있다.

> **핵심 교훈:** 겉으로 보이는 증상(스크롤이 끌려간다)과 진짜 원인(store가 스트림을 재시작시킨다)이 다른 레이어에 있을 수 있다. 세 번의 가설이 전부 "스크롤 코드"에 머물러 있었던 건, 증상이 스크롤로 보였기 때문에 스크롤 코드만 팠기 때문이다. **"무엇이 이 상태를 만들었는가"를 증상이 나타난 레이어보다 한 단계 위(스크롤 ← messages ← store)에서 물었어야** 더 빨리 좁혀졌을 것.

### 다음 세션에서 할 일

- `chatStore`의 `connect`, `sendMessage`, `retryLastMessage`, `regenerateMessage` 네 진입점에 로그를 심어 25초 뒤 정확히 무엇이 재호출되는지 확정
- 유력 후보: `connect()`의 에러 복구 경로(`getMessages` 실패 시 `retryLastMessage()` 자동 호출) — 그런데 정상 흐름에서 왜 25초 뒤 이게 도는지는 아직 미확인
- 원인 확정 후에만 스크롤 쪽 코드(가설 1~3에서 추가한 방어 로직)를 정리 — 지금은 진짜 원인이 아닌 코드에 방어 로직만 세 겹 쌓인 상태라, 원인 잡은 뒤 불필요한 것들 걷어내야 함

## 오늘의 회고

- 반나절을 스크롤 코드에 썼는데 결과적으로 헛다리였다. **디버그 로그를 코드에 직접 심어 사실을 확인하는 접근**(추측 대신 관찰) 자체는 옳았지만, 관찰 범위를 스크롤 컴포넌트 안으로만 좁힌 게 시간을 잡아먹었다.
- 사용자가 "휠 굴리는 동안엔 괜찮은데 멈추면 밀린다"는 타이밍 단서를 정확히 짚어준 게 결정적이었다. 증상을 정확히 재현하고 관찰한 사람의 디테일이 로그보다 빠른 단서가 될 때가 있다.
- 다음엔 "이 상태(streaming)가 왜 지금 이렇게 됐는가"를 스크롤 effect보다 먼저, store의 액션 진입점에서 확인하고 시작할 것.
